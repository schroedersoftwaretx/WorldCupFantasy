# World Cup Fantasy

This is **Phase 1** of the 2026 FIFA World Cup fantasy draft project: the data
spine. It provides a Postgres schema, an internal `StatsProvider` abstraction
over [api-sports.io / API-Football v3](https://www.api-football.com/), and an
idempotent ingestion CLI.

No scoring, league, draft, lineup, auth, or UI code lives here. Phases 2â€“5
build on top of this layer; the contracts here (`StatsProvider`, `stat_line`
immutability) are deliberately stable so that scoring can be recomputed any
time from raw stat records.

## Layout

```
src/
  data/
    db/          Drizzle schema + Postgres client
    provider/    StatsProvider interface + ApiFootballProvider + FixtureMockProvider
    ingest/      Idempotent ingestion services (squads / schedule / fixture stats)
  cli/           CLI entry points
  migrate.ts    Apply Drizzle migrations against DATABASE_URL
drizzle/        Generated SQL migrations
test/
  fixtures/      Hand-crafted JSON shaped like real api-sports.io v3 responses
  unit/          Provider mapping unit tests
  integration/   Testcontainers-backed end-to-end ingest tests
```

Nothing outside `src/data/provider/` is allowed to import the vendor SDK or
construct vendor URLs. The provider boundary is what lets us swap data sources
and mock the live feed for tests.

## Migrations

The schema is declared in `src/data/db/schema.ts`. Migrations are generated by
drizzle-kit into `./drizzle/` and applied at runtime.

```bash
# Generate a new migration after editing the schema
npm run migrate:generate

# Apply pending migrations against $DATABASE_URL
npm run migrate
```

Tests spin up an ephemeral Postgres via Testcontainers and apply the same
migrations â€” there's no separate "test schema."

## Ingestion CLI

The CLI dispatches to the three Phase 1 ingestion subcommands. By default it
talks to the live provider (API-Football). Set `MOCK_FIXTURES_DIR` to point
the CLI at a directory of local JSON fixtures instead â€” this is how offline
development and tests run.

```bash
# Squads â†’ upsert national_team + player
node --env-file=.env --import tsx src/cli/index.ts ingest:squads

# Full 104-match schedule â†’ upsert fixture
node --env-file=.env --import tsx src/cli/index.ts ingest:schedule

# Per-player stats for a single finished match â†’ upsert stat_line rows
node --env-file=.env --import tsx src/cli/index.ts ingest:fixture-stats <sourceFixtureId>
```

Or via the `cli` script:

```bash
npm run cli -- ingest:squads
npm run cli -- ingest:schedule
npm run cli -- ingest:fixture-stats 12345
```

All three commands are **idempotent**. Running them twice in a row produces
identical database state; rows are upserted, and `stat_line` updates are
gated by a `source_revision` so corrections from the provider are detectable.

## Provider round â†’ stage enum mapping

API-Football encodes the tournament round as a free-form string on each
fixture. The mapping below is applied during `ingest:schedule`. Any unknown
round name causes the ingest to fail fast rather than silently mis-stage a
fixture.

| `fixture.round` (provider)        | Stage enum   |
|-----------------------------------|--------------|
| `Group Stage - 1`                 | `GROUP_1`    |
| `Group Stage - 2`                 | `GROUP_2`    |
| `Group Stage - 3`                 | `GROUP_3`    |
| `Round of 32`                     | `R32`        |
| `Round of 16`                     | `R16`        |
| `Quarter-finals`                  | `QF`         |
| `Semi-finals`                     | `SF`         |
| `3rd Place Final`                 | `THIRD_PLACE`|
| `Final`                           | `FINAL`      |

The mapping is centralized in `src/data/provider/api-football-mapping.ts` and
covered by unit tests; update both if the vendor changes a label.

## Scoring (Phase 2)

The scoring engine (`src/data/scoring/`) is a pure, recomputable function
over the immutable `stat_line` rows. The default ruleset matches section 5.1
of `WORLDCUP_FANTASY_PLAN.md` exactly; any change to a point value yields a
new `ruleset_version` (a content hash), and the `score_entry` table keeps
rows for every ruleset side by side so what-if rulesets do not wipe the
canonical one.

```bash
# Rebuild every score_entry row from stat_line under the default ruleset
npm run cli -- score:recompute

# Just one fixture (run this right after ingest:fixture-stats)
npm run cli -- score:recompute 8001
```

`score:recompute` is idempotent: rerunning it with no stat changes reports
`skipped` for every row.

## Leagues & rosters (Phase 3)

Phase 3 adds the private-group layer: managers, leagues, token-based
invites, and the roster validator.

```bash
# Register managers (identified by Firebase UID; JWT verification lands
# with the future API layer)
npm run cli -- manager:create <firebaseUid> "<display name>" <email>

# Create a league. The owner's membership + fantasy_team are created in the
# same transaction. The league embeds a full copy of the scoring ruleset.
npm run cli -- league:create <ownerFirebaseUid> "<league name>" [maxManagers]

# Invite + join (token-based)
npm run cli -- league:invite <leagueId> [email]      # prints a token
npm run cli -- league:join <token> <managerFirebaseUid>

# Draft players onto a team; the roster validator runs on every add
npm run cli -- roster:add <fantasyTeamId> <sourcePlayerId>
npm run cli -- roster:show <fantasyTeamId>
```

### Roster rules (sections 4.2 / 4.3 of the plan)

A roster is 23 players: GK 2-4, DEF 6-8, MID 5-8, FWD 4-8. The maximums
double as the per-manager draft caps. The validator is pure and enforces
three things on every `roster:add`:

1. the position is not at its cap,
2. the player is not already drafted in that league (a unique
   `(league_id, player_id)` constraint backstops this in the DB), and
3. the pick keeps a legal 23-man roster **reachable** - it will refuse a
   pick that strands a position minimum. This completability check
   (`isRosterCompletable`) is the primitive Phase 4's constraint-aware
   autopick is built on.

## Async draft & autopick (Phase 4)

Phase 4 adds the notification-driven snake draft. It is deliberately not a
live room: managers pick whenever they like within a per-pick timer (12h
per the plan), and a scheduled tick autopicks for anyone who lapses.

```bash
# One draft room per league; shorten the timer for a fast cycle
npm run cli -- draft:create <leagueId> [pickTimerHours]
npm run cli -- draft:start <leagueId>          # freezes a random snake order

# A manager makes their pick (turn-checked against the snake order)
npm run cli -- draft:pick <leagueId> <fantasyTeamId> <sourcePlayerId>

# Run this on a schedule (cron, etc.) - it autopicks every expired pick
npm run cli -- draft:tick

npm run cli -- draft:status <leagueId>
npm run cli -- player:rank <sourcePlayerId> <rank>   # seed the autopick board
```

### How it works

- **Snake order.** Round 1 runs the order forwards, round 2 backwards, and
  so on. The math is pure (`src/data/draft/snake.ts`).
- **The 12-hour timer.** Each pick has a deadline. `draft:tick` calls
  `processExpiredPicks`, which autopicks any pick whose deadline has
  lapsed. It is idempotent - a tick with nothing expired is a no-op.
- **Constraint-aware autopick.** The autopick takes the best available
  player (lowest `draft_rank`) that is still a *legal* pick: under the
  position's draft cap **and** leaving a legal 23-man roster reachable.
  That second test (`isRosterCompletable`, from Phase 3) is what stops the
  autopick from drafting a manager into a corner.
- **Atomic picks.** A pick, its roster slot, and the draft advance commit
  in one transaction - the draft can never be left half-advanced.
- **Durable notifications.** Every notification is written to
  `draft_notification` (status PENDING) before delivery is attempted, then
  marked SENT/FAILED. A FAILED or undelivered row is retried on the next
  tick, so a manager is never silently left un-notified. The `Notifier`
  interface (`RecordingNotifier`, `ConsoleNotifier`) is where a real SMTP
  implementation drops in.
- **Completion.** The 552nd (or Nx23rd) pick marks the draft COMPLETE and
  advances the league SETUP -> DRAFTING -> ACTIVE.

## Best-ball scoring & standings (Phase 5)

Phase 5 closes the MVP: the best-ball lineup optimizer and league
standings. Both are pure, recomputable views over `score_entry` - nothing
is stored. Because `score_entry` is itself rebuilt whenever stats ingest,
recomputing standings on demand means they are always current; that is the
"live updates" story, with no caching layer.

```bash
npm run cli -- standings:show <leagueId>            # ranked table
npm run cli -- standings:show <leagueId> --periods  # + per-period formations
```

### Best-ball

There is no lineup deadline. For each of the nine scoring periods (the
tournament stages) the optimizer retroactively picks the highest-scoring
legal starting XI from the manager's 23-man roster. A legal XI is 1 GK
plus one of four formations - **4-3-3, 4-4-2, 5-2-3, 5-3-2** - derived
from the section 4.1 ranges (DEF 4-5, MID 2-4, FWD 2-3). For a fixed
formation the positions are independent, so the optimum is simply the
best formation among the four, each filled with the top-scoring players
per position. A team's total is the sum of its nine period scores.

### Standings & tie-breakers

Teams are ranked by the section 5.3 ladder: total points, then points by
all rostered players in the Final match, then tournament goals, then
assists; a full tie shares the rank.

|---------------------------------------|---------------------------------------------|
| `src/data/standings/lineup.ts`        | Formations + the pure best-ball optimizer.  |
| `src/data/standings/standings.ts`     | `computeStandings` + the tie-breaker ladder. |
| `test/unit/standings.test.ts`         | Optimizer + ranking unit tests.             |
| `test/integration/standings.test.ts`  | Standings over real rosters + score_entry.  |


## Web app: Next.js shell & API boundary (W1)

W1 is the first phase of the web build (see `WEBAPP_PLAN.md`): a Next.js app
that lives in this same repository and imports `src/data/` directly as a
library. There is no separate API service â€” at the two-friends scale the
Next.js server calls the backend functions in-process. Game logic is never
reimplemented; the web layer is wiring and screens over a backend that is
already built and tested.

```bash

npm run dev            # http://localhost:3000

# Production build / serve
npm run build
npm start
```

### What W1 ships

A representative slice that proves the full stack â€” browser -> Next.js ->
`src/data/` -> Postgres â€” end to end.

**Route handlers** under `app/api/`, each returning a consistent JSON
envelope (`{ ok: true, data }` on success, `{ ok: false, error: { message,
code } }` on failure):

| Route | What |
|---|---|
| `GET /api/health` | Liveness probe; runs `select 1` (200 if up, 503 if down). |
| `GET /api/leagues` | Every league with its current member count. |
| `GET /api/leagues/[leagueId]` | One league with its members (404 if missing). |
| `GET /api/standings/[leagueId]` | Live standings, recomputed on every request. |

**Pages** (App Router server components): a landing page listing the
leagues, and a per-league standings read view that calls `computeStandings`

The route handlers map the backend's typed domain errors
(`LeagueError` / `RosterError` / `DraftError`) to HTTP 400, an explicit

W1 is **unauthenticated** â€” every league is visible. Sign-in and

write endpoints (create league, invite, draft) are added in W3â€“W5.

## Web app: Authentication (W2)

W2 puts a sign-in wall in front of the app. Authentication is **Google
sign-in via Firebase**; a manager's identity is their Firebase UID, stored in
the `manager.firebase_uid` column the schema has carried since phase 3.

**Sign-in requires a Firebase project.** Create one and fill in seven `.env`
values by following `FIREBASE_SETUP.md` â€” it is a ten-minute, one-time setup.
Until then the app runs but every page redirects to a `/login` screen that
explains what is missing.

### How it works

```
  Browser  --(Google popup)-->  Firebase  --(ID token)-->  Browser
  Browser  --POST /api/auth/session { idToken }-->  Server
  Server   verifies the ID token (Admin SDK), mints a 5-day session cookie
  Server   --Set-Cookie: wc_session (httpOnly)-->  Browser
  ...every later request...
  Browser  --wc_session cookie-->  Server
  Server   verifies the cookie, find-or-creates the manager by firebase_uid
```

- **Session cookies, not bearer tokens.** The short-lived Firebase ID token
  is exchanged once for a long-lived, httpOnly session cookie. Server
  Components (the leagues and standings pages) and route handlers both read
  it directly â€” no client-side token juggling.
- **The gate is enforced server-side.** Edge middleware does a cheap
  cookie-presence check and redirects signed-out visitors to `/login`. The
  real check is in every page (`getCurrentUser`, then `redirect`) and every
  data route (`requireUserForRoute`, a 401 envelope) â€” both verify the cookie
  with the Firebase Admin SDK on the Node runtime. The `/api/health` probe is
  intentionally left open.
- **No registration step.** `createManager` is an idempotent upsert keyed on
  `firebase_uid`, so a first-ever sign-in provisions the manager row
  automatically.
- **Graceful when unconfigured.** With no Firebase env vars, token
  verification simply fails closed â€” every visitor is treated as signed out
  and sent to `/login`, which shows the setup message.

## Web app: Leagues UI (W3)

W3 is the first set of screens a signed-in user actually works with: a
dashboard of their leagues, a create-league form, and the invite flow. After
W3 two people can reach the same league entirely through the browser.

### What W3 adds

- **A dashboard scoped to you.** `/` lists only the leagues you belong to
  (W1 listed every league - that was the pre-auth placeholder). A brand-new
  user sees an empty dashboard with the create form.
- **Create a league.** A form on the dashboard `POST`s to `/api/leagues`;
  the backend `createLeague` makes the league, your OWNER membership, and
  your fantasy team in one transaction. You land on the new league page.
- **A league overview page.** `/leagues/[leagueId]` shows the managers, your
  role, a link to standings, and - for the owner - the invite panel.
- **The invite flow.** The owner generates an open invite link
  (`/invite/[token]`); anyone who opens it, signs in, and clicks "Join" is
  added to the league. The backend `acceptInvite` enforces every rule
  (pending, not expired, not full, not already a member).
- **Membership gates.** Every league page and data route now checks that you
  are a member - a non-member gets a not-found response that does not even
  confirm the league exists. Generating invites is owner-only.
- **Deep-link-safe sign-in.** Opening an invite link while signed out
  bounces through `/login?next=...` and returns you to the invite after you
  sign in. The `next` target is validated (`safeNextPath`) so it can only
  ever be a path on this site, never an open redirect.

### Flow

```
  Owner   /  -> create league -> /leagues/3
  Owner   /leagues/3 -> Generate invite link -> /invite/<token>
  Owner   sends the link to a friend
  Friend  opens /invite/<token>  -> /login?next=/invite/<token> -> signs in
  Friend  back on /invite/<token> -> Join this league -> /leagues/3
  Both    /leagues/3 now lists both managers
```

## Web app: The draft room (W4)

W4 is the centerpiece - the screen where a draft actually happens, browser to
browser. It is the only phase with substantial new interactive UI.

### What W4 adds

- **The draft room** at `/leagues/[leagueId]/draft`. It renders the right
  view for the draft's state and refreshes by polling every 5 seconds:
  - **No draft yet** - the owner sees a create form (with a pick-timer field).
  - **Created, not started** - the owner sees a Start button (enabled once at
    least two managers have joined).
  - **In progress** - the live room: who is on the clock with a deadline
    countdown, the snake order, a searchable / filterable board of available
    players, the viewer's roster with live position counts, and the picks log.
  - **Complete** - the final picks, with a link to standings.
- **A turn-checked, roster-validated pick.** Drafting a player POSTs to the
  pick route; the backend `makePick` rejects a pick that is out of turn or
  would break the 23-man roster rules. The board greys out the Draft button
  for players that are not a legal addition to the viewer's roster.
- **The draft tick.** `processExpiredPicks` autopicks any pick whose timer has
  lapsed. It runs two ways: a scheduled Vercel Cron call to
  `/api/cron/draft-tick` (see `vercel.json`; protected by `CRON_SECRET`), and
  a manual "Process timeouts" button in the room.
- **Owner controls.** Creating and starting the draft are owner-only; making
  a pick is the on-the-clock manager; the manual tick is open to any member
  (it only ever acts on already-expired picks).

### The endpoints

| Route | What |
|---|---|
| `GET /api/leagues/[id]/draft` | The polled draft state. |
| `POST /api/leagues/[id]/draft` | Create the draft room (owner). |
| `GET /api/leagues/[id]/draft/board` | The available-player board. |
| `POST /api/leagues/[id]/draft/start` | Freeze the order, start (owner). |
| `POST /api/leagues/[id]/draft/pick` | Draft a player (on-the-clock manager). |
| `POST /api/leagues/[id]/draft/tick` | Manual timeout processing. |
| `GET /api/cron/draft-tick` | Scheduled tick across all drafts. |

The state poll and the player board are separate endpoints on purpose: the
board is heavier, so it is fetched on load and only re-fetched when a pick
has been made, while the lightweight state is polled continuously.
